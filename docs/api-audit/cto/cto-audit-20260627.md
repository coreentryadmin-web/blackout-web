## EXECUTIVE SUMMARY

The platform is **architecturally sound and unusually well-hardened** — prior audits closed the pg-pool-no-error-handler crash, Redis `family:0`, bounded LRU caches, constant-time cron auth, layered AI spend gates, and Polygon WS leader election. The codebase is not the liability. **The liabilities are three:**

1. **The flagship's durable record is empty in production.** `spx_play_outcomes` has **0 rows all-time** and `spx_signal_log`'s last write was **2026-06-17 (10 days stale)**. The SPX engine evaluates and caches to Redis but its durable persistence has produced no gradeable outcomes ever — so the launched SPX Slayer's P&L / track-record panels are empty against live ground truth. The optionTicket veto was neutered (`spx-play-config.ts:405`), but opens still aren't reaching the durable store; the mechanism is the silent close-without-outcome + swallowed `recordPlayEntry` write path (P0-1/P0-2 below).

2. **The UW WebSocket has zero multi-replica protection.** Polygon got leader election; UW did not. At 5 replicas every replica opens its own UW multiplex socket → 5× join traffic against a **2-RPS cluster budget**, plus unbounded fire-and-forget persist fan-out (`uw-socket.ts`, no `p-limit`). This is the thing that breaks first when you scale horizontally.

3. **`reconcileAllMemberships` is O(users) serial network I/O with no lock.** Tier = entitlement = money. The hourly cron has no advisory lock and no concurrency bound; at real scale it runs for minutes past timeout and, if it overlaps itself or fires on multiple replicas, races Clerk writes and hammers Whop's rate limit.

**What breaks first at 10×:** the UW socket fan-out (item 2) — it's load-bearing, always-on during RTH, and has no guard at all. Single biggest single-PR win in the whole audit is porting Polygon's Redis-SETNX leader pattern to UW.

**Overall technical health: B+.** Strong engineering, well-defended single-process. The risks are at the *distributed-systems seams* (multi-replica WS/reconcile) and in *durable-write correctness* on the flagship, not in obvious bugs.

## CRITICAL FINDINGS (Fix Before Next Deploy)

### P0-1 — SPX flagship durable track record is empty in prod (live-confirmed)
**Evidence (live DB, 2026-06-27):** `spx_play_outcomes` = 0 rows / 0 closed all-time; `spx_open_play` = 0 rows; `spx_signal_log` = 12 rows all-time, **last write 2026-06-17T17:36Z**. The track-record UI hammers the empty table: **16,693 seq scans** recorded on `spx_play_outcomes`.
**Mechanism (code):**
- `db.ts:1234-1238` — `insertOpenSpxPlay` runs `UPDATE spx_open_play SET status='closed', closed_at=NOW() WHERE session_date=$1 AND status='open'` **before** the insert, with **no exit price, no outcome row**. A force-closed play is invisible to the track record.
- `spx-play-engine.ts:915-927` — after a successful `openPlay()`, `recordPlayEntry` is wrapped in a `try/catch` that only `console.error`s. If it throws, the play is **live and managed** but `spx_play_outcomes` never gets an entry, so `recordPlayCloseOutcome` later returns `rowCount 0` (`db.ts:1454`). A real win/loss is shown as "never happened."
**Impact:** Direct violation of the live+correct+grounded mandate on the launched flagship — members see an empty/wrong P&L. **P0 (correctness of money-facing data).**
**Caveat:** Today is Saturday; zero opens *today* is expected. The damning fact is **zero outcomes across the engine's entire life** + a 10-day-stale signal log. **Action: on the next RTH session, confirm (a) plays now open post-veto-neuter, and (b) an outcome row is written on open AND close.** If opens still don't persist, the bug is upstream of the veto.

### P0-2 — `recordPlayEntry` write failure is swallowed → untracked live position
`spx-play-engine.ts:915-927`. Same root as P0-1, called out separately because it persists *after* any veto fix. Even once opens start, a fraction will open with no outcome row. Fix: make the entry write part of the same transaction as the open, or fail the open if the entry can't be recorded (fail-closed on durability).

### P0-3 — `coaching/alerts` POST fails OPEN if `CRON_SECRET` is unset
`src/app/api/coaching/alerts/route.ts:36-40`. The only cron-writer not using the hardened `isCronAuthorized`. Two defects: (1) `if (cronSecret && auth !== \`Bearer ${cronSecret}\`)` — when the env var is empty the guard is **skipped entirely**, allowing unauthenticated INSERTs into `coaching_alerts`; (2) plain `!==` is **not constant-time** (timing leak). Fix: replace with `if (!isCronAuthorized(req)) return 401;`. **P0 because it can become an open public write endpoint under a single missing env var** (a recurring failure class on this platform).

### P0-4 — UW WebSocket has NO multi-replica guard
`src/lib/ws/uw-socket.ts` (entire `UwSocketManager`, `init-data-sockets.ts:44`). Grep for `leader|SETNX|REPLICA` returns zero matches. At 5 replicas: 5 independent UW multiplex sockets, each firing 6 channel joins on every (re)connect → ~30 join frames in a reconnect storm against a **2-RPS cluster cap**; 5× DB insert-attempt load (DB rows saved by ON-CONFLICT, but the work is still 5×'d); 5× Discord/SSE fan-out. **P0 at horizontal scale** (currently masked — prod appears to run ~2 replicas per pool-usage probe). Fix: port `polygon-socket.ts:117-156` Redis-SETNX leader lock to UW; non-leaders read stores from Redis.

## REAL-TIME / WEBSOCKET HEALTH

| Socket | Reconnect (backoff+jitter+cap) | Error handler | Multi-replica safe | Memory safe | Top issue |
|---|---|---|---|---|---|
| `polygon-socket.ts` | ✅ exp, jitter `:179`, cap 60s | ✅ `:393` | ✅ Redis leader `:117-156` | ✅ | P1-1 lock-refresh TOCTOU; P1-4 wedge-on-construct-fail |
| `uw-socket.ts` | ✅ exp, cap 30s | ✅ `:292` | ❌ **none** | ⚠️ unbounded persist fan-out | **P0-4** + P1-2 backpressure |
| `options-socket.ts` | ✅ exp, jitter `:297`, cap 60s | ✅ `:355` | ⚠️ per-replica cap only (gated off) | ❌ `optionMarks` leak `:443` | P1-3 leak; inherits P0-4 when enabled |

**P1 details:**
- **P1-1** `polygon-socket.ts:149-154` — leader-lock refresh uses bare `expire(KEY,25)` with no ownership check and a constant `"1"` value (`:138`). A GC/CPU-throttle pause past 25s lets replica B acquire the lock while A's refresh blindly extends B's key and A still believes it's leader → **two live Massive index sockets**, loser churns `code=1008`. Fix: tag lock with a per-process token, refresh via Lua compare-and-extend.
- **P1-2** `uw-socket.ts:592-615` — `flow_alerts` handler does `void persistAndPublishFlowAlert(...)` per alert with **no concurrency bound**. Open-bell burst → unbounded pending promises on the heap if DB/Discord slow. Fix: semaphore + drop/coalesce on saturation.
- **P1-3** `options-socket.ts:443` — `optionMarks.set(occ,...)` is never deleted on `unsubscribe` (`:565-576`). Monotonic growth across expiries over a long-lived replica. Redis layer has TTL; in-memory leaks. Fix: delete on unsubscribe or sweep stale.
- **P1-4** `polygon-socket.ts:405-409` — connect-throw doesn't release the leader lock / `indicesIsLeader` flag. A replica with a persistently-failing WS construct **holds the lock forever and delivers no frames**, starving the whole cluster's index feed. Fix: release lock after N consecutive construct failures.

Heartbeat note: none of the three do app-level ping/pong with pong-timeout tracking; all rely on data-liveness watchdogs (Polygon 25s tight; UW 75s stall; Options 5 min but degrades safely via 30s null → REST). Acceptable, but a quiet-minute half-open on UW relies entirely on the 75s stall timer.

## CONCURRENCY RISKS

- **P1-1 (race)** `spx-desk-merge.ts:82-220` — module-singleton `lastGoodStructure`/`lastGoodStructureAt` is read-modify-written on **every** `mergePulseIntoDesk` call. `mergeDeskLayers` runs per-request on the server loader, so concurrent requests with different VWAP/HOD **cross-contaminate** — request A's response can carry request B's structure levels. For a "100% correct vs source" surface this is a real grounding hole. Fix: scope structure state per-request or key it immutably by session snapshot.
- **P1-2 (race)** `spx-desk-merge.ts:186-201` — `seedStructureCacheFromBase` fires `void loadStructureFromRedis()` then synchronously seeds from `base` in the same tick; the async Redis load resolves a cycle too late, so the first request after a worker boot serves degraded (possibly null) structure even when Redis had good values.
- **P2 (race)** `polygon-options-gex.ts:1218-1254` — `appendGexEodSnapshot` read-modify-writes the EOD series with no CAS; at the `GEX_EOD_MAX=10` trim boundary two cron replicas can drop different tail entries → per-replica day-over-day deltas diverge.
- **Module-level mutable state:** ~80 reviewed across `src/lib`; all are connection singletons, caches, or rate-limiter state **except** the `spx-desk-merge` structure singletons above. No per-user data in module scope.
- **Promise.all:** no critical writer relies on un-settled `Promise.all` where one rejection silently drops siblings — the Anthropic tool-loop, Discord fan-out, and coaching INSERT batch all catch per-item or wrap in try/catch. Clean.
