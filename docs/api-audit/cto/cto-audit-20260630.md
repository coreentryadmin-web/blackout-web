## EXECUTIVE SUMMARY

**Overall technical health: B+ / A-.** This remains a maturely-defended codebase ÔÇö per-route self-guarded auth with constant-time secret compare, cluster-aware Redis-Lua rate limiters, mandatory pg/Redis `error` handlers (no replica-crash gap), prompt-cache discipline, WS reconnect/backoff/leader-election on the primary feed, and a money path (`insertOpenSpxPlay`) triple-protected by advisory lock + unique partial index + 23505-retry-in-transaction. The concurrency pass found **no P0 data-corruption bug**.

**The single biggest theme this run is "the cliff moved, nobody re-surveyed."** Two prior recommendations were actioned in prod env: `REPLICA_COUNT` is now `5` (was unset Ôćĺ the 06-29 C1 rate-limit P0 is **resolved**) and `PG_POOL_MAX` is now `15` (was 5). Both were correct fixes. But each silently relocated a scale risk that no one re-measured:

1. **`REPLICA_COUNT=5` activated the dormant WS leader-election gap.** The UW and options sockets have no cross-replica leader election (only Polygon does). With 5 real replicas, that is **5 duplicate UW connections and 5 options connections cluster-wide today** ÔÇö the options socket's own comments warn this triggers a `code=1006` boot-each-other reconnect storm. This is tasks #102/#104 and it is now *active*, not theoretical.
2. **`PG_POOL_MAX=15` ├Ś 5 replicas = 75 backend connections against an effective ceiling of 97** (`max_connections=100` Ôłĺ 3 superuser-reserved). The live probe found **no PgBouncer** (empty `application_name`, direct `client backend` connections) ÔÇö so `db.ts:92`'s comment "PgBouncer sits in frontÔÇŽ handles real connection pooling" is **not true in this environment**. The app pool *is* the backend connection count. Add 19 cron services + held-advisory-lock clients and the cluster can plausibly hit `FATAL: too many connections` at RTH peak.

**What breaks first at 10├Ś traffic:** the Postgres connection ceiling (75/97 Ôćĺ exhaustion), accelerated by **per-call telemetry INSERT amplification** ÔÇö `api_telemetry_events` grew **619KÔćĺ822K rows / 314MBÔćĺ460MB in the ~27 hours between this report and the last**, driven by ~200K INSERTs/day (the code comment assumes ~30K/day ÔÇö a **5ÔÇô8├Ś under-estimate**), each a single un-batched write competing for the same pool that serves users.

**The one component that can still OOM a replica today** is unchanged from 06-29: `account/positions/stream` SSE remains the only one of four SSE routes with **no connection cap and no backpressure** ÔÇö flagged 06-29 as C2, **still unfixed**.

**Money/data risk:** `DAILY_AI_SPEND_KILL_USD` is **unset in prod** ÔÇö the org-wide AI hard kill-switch is disarmed (confirmed live). Spend is otherwise well-gated, but the catastrophic backstop is off.

## CRITICAL FINDINGS (Fix Before Next Deploy)

### C1 ÔÇö Postgres connection ceiling: 5 replicas ├Ś pool 15 = 75 of 97 usable, no PgBouncer in front
**Location:** `src/lib/db.ts:92-108` (pool config + the inaccurate PgBouncer comment); live env `PG_POOL_MAX=15`, `REPLICA_COUNT=5`; live `pg_settings`: `max_connections=100`, `superuser_reserved_connections=3`.
**Evidence:** Live `pg_stat_activity` shows only `client backend` connections with empty `application_name` and distinct client addrs ÔÇö **no PgBouncer process is interposed.** Railway's standard Postgres template ships without a bouncer; the `db.ts:92` comment is aspirational, not factual here.
**Repro / math:** At RTH peak each replica saturates its 15-slot pool (SSE polling + telemetry INSERTs + user reads + warming crons). 5 ├Ś 15 = **75**. The 19 cron *services* each open their own short-lived pool on fire; migrations hold a dedicated lock-client; `heldLockClients` (`db.ts:754`) pins one connection per held advisory lock (spx-eval, flow-ingest) for its whole lifetime. Effective ceiling is 97. The gap (22) is thin and shared with everything non-web. Once hit, new checkouts get `FATAL: sorry, too many connections` Ôćĺ that replica throws on **every** DB query Ôćĺ 503 cascade for the users it serves.
**Why P0:** availability of the core paid product, self-inflicted, and the headroom shrank precisely *because* `PG_POOL_MAX` was raised without re-checking cluster-wide math.
**Fix (pick one, in order):** (a) **stand up PgBouncer** in transaction mode (Railway add-on or sidecar) and point `DATABASE_URL` at it ÔÇö then 15-per-replica multiplexes safely and the `db.ts` comment becomes true; (b) failing that, **lower `PG_POOL_MAX` to ~10** (5├Ś10=50, safe) and rely on queueing; (c) raise `max_connections` only if the Postgres plan's RAM supports it (each backend Ôëł 5ÔÇô10 MB). Also: correct or delete the false PgBouncer comment at `db.ts:92` so the next engineer doesn't trust it.

### C2 ÔÇö `positions/stream` SSE still has no connection cap and no backpressure Ôćĺ replica OOM  *(unfixed since 06-29)*
**Location:** `src/app/api/account/positions/stream/route.ts` ÔÇö verified this run: `grep` for `MAX_STREAMS|activeStreams|sseBackpressure|desiredSize` returns **nothing**.
**Repro:** Backgrounded mobile tabs (stalled TCP) or many open Night's Watch tabs. Each connection spawns a 15s heartbeat `setInterval` + a self-rescheduling 3s push loop and calls `send()` with no `controller.desiredSize` check and no `MAX_STREAMS` gate. A dead client's controller queue grows one full positions-JSON every 3s, unbounded. The other three SSE routes (`pulse`, `flows`, `admin/apis`) all have the cap + `sseBackpressureExceeded` drop; this is the lone outlier.
**Why P0:** the only identified path that can exhaust a replica's memory/fds under normal user behavior.
**Fix:** port the `activeStreams` counter + `MAX_STREAMS` 503 gate + `sseBackpressureExceeded(controller.desiredSize)` drop from `pulse/stream`, and add a `cancel()` handler on the ReadableStream (today cleanup relies solely on `req.signal` abort ÔÇö add `cancel()` as the second teardown path). ~30 minutes; the pattern is copy-paste from the sibling route.

### C3 ÔÇö AI hard kill-switch disarmed in production
**Location:** `src/lib/providers/anthropic.ts:319` (`isAiSpendCeilingTripped`, gates every `anthropicText`/`anthropicToolLoop`); live env: **`DAILY_AI_SPEND_KILL_USD = <UNSET>`**.
**Impact:** The org-wide daily spend ceiling is opt-in and inert until the var is set. All the *other* gates (Largo per-user concurrency + daily query budget, shared-cache collapse, global concurrency ceiling) are healthy and make a per-user drain implausible ÔÇö but there is no hard backstop against an unforeseen runaway (a new endpoint, a retry storm, a prompt-size regression).
**Why P0-adjacent (money):** a single missing env var is the only thing between "well-gated" and "no ceiling." One-line fix with outsized protection.
**Fix:** set `DAILY_AI_SPEND_KILL_USD` to a sane daily cap (e.g. 2ÔÇô3├Ś observed daily spend) in the blackout-web Railway service. Also pin `UW_GLOBAL_MAX_RPS=2` and `POLYGON_GLOBAL_MAX_RPS` ÔÇö both are currently **unset**, relying on code defaults (UW default is 2, which is correct, but an explicit pin removes the "someone bumps the default" risk).
