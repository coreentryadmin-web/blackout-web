## EXECUTIVE SUMMARY

**Overall technical health: B+. This is a mature, defensively-written codebase** — prior pentests are visible in the code, auth is self-guarded per-route, rate limiters are cluster-aware via Redis Lua, the pg Pool and Redis clients both have the mandatory `error` handlers that prevent replica crashes, prompt-cache and output-cache discipline is real, and WS reconnect/backoff/leader-election is present on the primary feed. **Several memory notes are now stale (resolved): the `db.ts` pool error-handler gap is closed (db.ts:113), Polygon DOES have a rate limiter, and Anthropic prompt caching IS implemented.**

**The single biggest risk right now is a configuration gap, not a code bug:** `REPLICA_COUNT` is unset in committed config (defaults to `1`), which silently breaks the UW rate limiter's degraded-mode math. **What breaks first at 10x traffic** is the shared 5-connection-per-replica Postgres pool, starved by long-lived SSE polling + per-call telemetry INSERT amplification + occasional unbounded scans. **The one component that can actually OOM a replica today** is the `positions/stream` SSE route, which — alone among the four SSE routes — has no connection cap and no slow-client backpressure.

## CRITICAL FINDINGS (Fix Before Next Deploy)

### C1 — `REPLICA_COUNT` unset → cluster blows past the UW rate-limit ceiling on any Redis blip, with the alarm suppressed
**Location:** `src/lib/providers/uw-rate-limiter.ts:18,30-33,131-152,196-198` + Railway service env (missing var)
**Repro:** With ≥2 replicas, induce a Redis outage (or cold-start window). Each replica's `effectiveMaxRps()` falls back to `DEGRADED_LOCAL_RPS = UW_GLOBAL_MAX_RPS / REPLICA_COUNT`. `REPLICA_COUNT` defaults to `1`, so each replica computes `2 / 1 = 2` RPS and the cluster emits `2 × N` RPS. At 5 replicas that is **10 RPS against a 2-RPS cluster ceiling → 429 storm → reactive breaker trips cluster-wide → blank SPX desk / flows for every user.** Worse: the degraded-state ops alert (`alertRedisDegradedOnce`, line 131-152) only fires when `REPLICA_COUNT > 1`, so with the default of 1 the cluster overshoots **silently**.
**Why it's P0:** an availability outage of the core paid product (desk + flows), self-inflicted, undetected. The same `REPLICA_COUNT` dependency exists in `polygon-rate-limiter.ts` (smaller blast radius — Massive is more permissive).
**Fix:** set `REPLICA_COUNT` to the real replica count in the Railway service env (and pin `UW_GLOBAL_MAX_RPS=2`, `POLYGON_GLOBAL_MAX_RPS`). Better: derive replica count at boot instead of trusting a hand-set env. This is a one-line env change with outsized protective value.

### C2 — `positions/stream` SSE route has no connection cap and no backpressure → replica OOM
**Location:** `src/app/api/account/positions/stream/route.ts:20-92`
**Repro:** Open the Night's Watch positions stream in many tabs, or let mobile clients background their tab (stalled TCP). Each connection spawns a 25s heartbeat `setInterval` (line 75) + a self-rescheduling 3s push loop (line 64) and calls `send()` (line 46) with **no `controller.desiredSize` / backpressure check** and **no `activeStreams`/`MAX_STREAMS` gate**. A slow/dead client's controller queue grows by one full positions-JSON payload every 3s, forever. The other three SSE routes (`pulse/stream`, `flows/stream`, `admin/apis/stream`) all have the cap + `sseBackpressureExceeded` drop; this one is the outlier.
**Why it's P0:** the only identified path that can exhaust replica memory/file-descriptors under normal user behavior (backgrounded tabs).
**Fix:** port the `activeStreams` counter + `MAX_STREAMS` 503 gate + `sseBackpressureExceeded(controller.desiredSize)` drop from `pulse/stream`, and wire the decrement into the existing `cleanup()` (line 40) so the cap accounting is complete. The abort listener (line 69) is already correct — only the cap + backpressure are missing.

**Hot-row churn / autovacuum (from `pg_stat_user_tables`):** several tiny, hot tables show extreme dead-tuple ratios and seq-scan counts — autovacuum isn't keeping pace with the write churn:
- `platform_meta` — **142,030 seq scans**, 10 live rows, **220% dead tuples**. This KV/heartbeat table is read and UPDATE-ed on a hot path; at 10 rows the seq scan is cheap, but the churn signals a frequently-rewritten row that becomes a **row-lock contention point under concurrency**. Watch it.
- `user_positions` — **200,419 seq scans**, 2 live rows, **1350% dead tuples** (idx used only 799×). The planner picks seq scan because the table is tiny *now*; the moment positions grow, those 200K-pattern scans stop being free. Performance cliff in waiting.
- `nighthawk_jobs` 720% dead, `nighthawk_dossiers_staging` 4000% dead, `nighthawk_editions` 300%, `lotto_plays` 125%, `users` 164%.
- **Action:** set per-table `autovacuum_vacuum_scale_factor`/`_threshold` aggressively low on the hot churned tables; they're small so vacuum is cheap and frequent is fine.

**Table growth / retention:** `api_telemetry_events` at **619K rows / 314 MB** dominates. It is indexed and read via index (825K idx-scans vs 34 seq), and `db-cleanup` prunes it (retention floor 90d, `db-cleanup-targets.ts:7`), but the write rate (one row per provider call, C-table item above) makes this the fastest-growing object. Confirm the nightly prune is actually running and the 90-day floor is intended for a table at this write rate.

**Missing FK indexes (all currently 0-row playbook tables, low priority):** `play_candidates.scan_run_id`, `play_cards.candidate_id`, `play_outcomes.play_card_id`, `playbook_tracked_positions.play_card_id` + `.playbook_run_id`. Add indexes before these tables get populated, or JOINs will seq-scan.

**Verified-correct DB (not flagged):** all queries parameterized (`$n` / `= ANY($1::text[])`) — no SQL injection; `db-cleanup` validates `${table}.${column}` against an allow-list before interpolation; `upsertNighthawkPlayOutcomes` is a single multi-row INSERT; `closeOpenPlay` / `appendLargoMessage` wrap their multi-writes in real transactions; advisory-lock helpers hold lock+unlock on the same dedicated client.

Per-replica transient over/under-count is harmless.

- **Telemetry in-memory ring buffer + `endpointStats` Map** — bounded (MAX_EVENTS 800, MAX_ENDPOINT_STATS 500 with LRU + endpoint templating to collapse per-symbol cardinality). No leak.
- **`Promise.all` usage** — the flagged sites in `nights-watch/enrichment.ts` are cache reads (WS marks/snapshots), not unguarded DB fan-out; no one-failure-fails-all data hazard found in the hot paths.

## 30-DAY RISK FORECAST

If nothing changes, in priority order of likelihood:

1. **UW 429-ban outage on a Redis blip (C1).** Redis cold-starts/blips are routine on Railway. The first one that hits while >1 replica is live emits N× the UW cap and trips the cluster breaker → blank desk/flows for all users, **with no alert** because the degraded-state alarm is gated off at `REPLICA_COUNT=1`. Single most likely 30-day incident. **Mitigation is a one-line env var.**
2. **Postgres pool starvation under load.** As concurrent users + the always-on telemetry INSERT amplification + SSE polling rise, the `max:5` pool gets pinned by a burst of slow `flow_alerts` JSONB scans → 15s connection-timeout → cascading 502s. Most likely to surface during a high-flow market open.
3. **`positions/stream` replica OOM (C2).** Needs only a handful of backgrounded mobile tabs to start unbounded controller-queue growth. Slower-burning than #1/#2 but a hard crash when it lands.
4. **`api_telemetry_events` unbounded growth** if the nightly prune ever silently fails (it's best-effort/try-caught) — 314 MB today, fastest-growing table; a stuck `db-cleanup` cron would let it balloon and slow every telemetry-adjacent query.
