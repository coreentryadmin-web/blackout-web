## P1 — Fix Today (Feature Broken or Degraded)

### P1-1 — admin/signal-analytics returns 500 (column does not exist) — **FIXED THIS RUN**
- **Evidence:** live Railway log `[admin/signal-analytics] error: column o.gates_blocked does not exist` (severity ERROR, routine errorMissingColumn).
- **Root cause:** the table `spx_signal_observations` defines the column `gates_blocked_json` (`src/lib/spx-signal-db.ts:28`), but `src/app/api/admin/signal-analytics/route.ts` referenced `o.gates_blocked` (no `_json` suffix) in BOTH queries:
  - gate-block-frequency query: `route.ts:197, 199, 200`
  - recent-observations query: `route.ts:263` (`SELECT ... gates_blocked`)
- **Impact:** the entire `/api/admin/signal-analytics` endpoint 500s on every call — admin signal-weight analytics fully broken. Admin-only, so not customer-facing (→ P1 not P0).
- **Fix applied (local commit, not pushed):** renamed `o.gates_blocked` → `o.gates_blocked_json` (lines 197/199/200) and `gates_blocked` → `gates_blocked_json AS gates_blocked` (line 263, alias preserves the TS row mapping at `route.ts:285`). tsc was clean before the edit and the change is inside SQL template strings, so types are unaffected.

### P2-1 — VAPID key pair is asymmetric → web push / gex-alerts still INERT (corrects prior run)

- **Impact:** web push requires BOTH keys — the public key is the client-side `applicationServerKey` for subscription; the private key signs server-side. With only the private key, no client can subscribe and no push is delivered. gex-alerts and all push notifications remain inert.

- **Fix:** set `VAPID_PUBLIC_KEY` to the matching public key of the existing private key (operator step). A half-configured pair is worse than none — it reads as "configured."
