## P1 — Fix Today (Feature Broken or Degraded)
- **P1-1 (DATA INTEGRITY · carried from 16:46 run · re-confirmed in code this run):**
  `src/lib/spx-play-outcomes.ts:177-178` unconditionally classifies any `STOP`,
  `THESIS`, or `was_loss` exit as `"loss"` **without checking `pnl_pts` sign** —
  unlike the THETA/SESSION branch (`:170-175`) and the TRAIL branch (`:183-186`),
  which both check the sign correctly. A `THESIS` exit can close **in profit** (you
  exit because the setup's thesis changed, not because price went against you), and
  that profitable trade is then stored `outcome='loss'`. **Live proof (16:46 run):**
  play #3 — long, `exit_action=THESIS`, entry 7432.13 → exit 7439.43, `pnl_pts=+7.30`
  — stored as `loss`; that day's A/A+ plays read 0/3 wins when the true figure is 1/3.
  **User impact:** understates win-rate and shows a winning trade as a loss on the
  track-record / P&L panels — the headline credibility surface for the SPX tool.
  **Fix:** for `STOP`/`THESIS`, treat `pnl_pts > 0` as a win (mirror the TRAIL
  branch); keep ≤ 0 as loss. One-liner, low risk. Not auto-fixed here because the
  audit cron's defined output is a report and the STOP-vs-profit semantics warrant a
  human nod (a hard stop-out is arguably always a loss; a trailing/THESIS exit is not).
